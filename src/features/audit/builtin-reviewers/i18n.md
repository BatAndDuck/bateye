Focus your review on:

## Hardcoded User-Visible Strings
- Raw string literals rendered directly in JSX/templates (e.g., `<p>Welcome back</p>`, `<button>Save changes</button>`) that should use a translation call (`t()`, `i18n.t()`, `$t()`, `useTranslation`)
- Error messages, validation messages, and toast notifications hardcoded in TypeScript/JavaScript that are displayed to the user without going through i18n
- Tooltip text, placeholder text, and `aria-label` values hardcoded as string literals instead of translated keys
- Email templates or notification payloads with hardcoded language strings
- Dynamic string construction using `+` or template literals that mixes translated and untranslated fragments, producing untranslatable composite strings

## Date and Time Formatting
- `new Date().toLocaleDateString()`, `toLocaleTimeString()`, or `toLocaleString()` called without a `locale` parameter — these fall back to the runtime locale which may not match the user's preference
- `date.toISOString()` or `.toString()` used in user-facing display contexts where locale-aware formatting is needed
- Hardcoded date format strings (e.g., `"MM/DD/YYYY"`, `"DD-MM-YYYY"`) rather than using `Intl.DateTimeFormat` or a library (dayjs, date-fns) with locale options
- Relative time expressions (e.g., "2 hours ago") computed manually without `Intl.RelativeTimeFormat`
- Timezone-insensitive date calculations that produce incorrect displays for users in different zones

## Number and Currency Formatting
- Hardcoded currency symbols (`$`, `€`, `£`) concatenated with numbers instead of `Intl.NumberFormat` with `style: "currency"` and `currency` options
- Numbers formatted with hardcoded thousand separators (commas) or decimal points that are locale-specific
- Percentage values formatted by manual multiplication and string concatenation rather than `Intl.NumberFormat` with `style: "percent"`
- Large numbers hardcoded without considering locale-specific digit grouping (e.g., Indian numbering system uses lakh/crore grouping)

## Pluralization and Grammar
- Plural forms handled with simple ternary (`count === 1 ? "item" : "items"`) instead of proper pluralization support (`i18nPlural`, `t('key', { count })`, `$tc()`)
- Missing plural rule coverage for languages with more than two plural forms (e.g., Russian, Polish, Arabic)
- Gender-inflected strings hardcoded rather than using grammatical gender support in the i18n library
- Ordinal numbers (1st, 2nd, 3rd) hardcoded in English rather than using `Intl.PluralRules` with `type: "ordinal"`

## String Concatenation and RTL
- Strings assembled by concatenation that embed translatable fragments in a fixed word order (e.g., `"Hello, " + name + "!"`) — translators cannot reorder words
- UI layout assumptions that break in RTL languages (e.g., hardcoded `text-align: left`, `margin-left`, `padding-left` without RTL overrides or logical properties like `margin-inline-start`)
- Directional icons (arrows, chevrons) that are not mirrored for RTL contexts
- String interpolation that places variable content at a fixed position that is grammatically wrong in other languages

## Locale Fallbacks and Configuration
- Missing locale fallback configuration — if a translation key is absent in the active locale, there is no fallback to a base language
- Translation function calls with no default/fallback value for keys that may not exist in all locale files
- Locale detection logic that defaults to `"en"` without checking browser `navigator.language` or user preferences
- Missing locale-specific locale files for a language variant that the application claims to support (e.g., `en-GB` vs `en-US`)

## Locale-Specific Assumptions
- Hardcoded sorting/collation using JavaScript's default array `.sort()` on user-facing strings rather than `Intl.Collator`
- Assumptions about name formatting (first name before last name) hardcoded in UI without locale-aware ordering
- Phone number or postal code input validation patterns hardcoded for a single country/region
- Address forms that assume a fixed set of fields (state/province, ZIP) rather than locale-adaptive address formats
